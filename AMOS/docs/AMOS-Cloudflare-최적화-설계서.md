# AMOS (BTC SMC + GAUGE-FLOW) Cloudflare 무료 티어 최적화 설계서

버전 1.2 (구현 반영판) · 대상 파일: `AMOS-Unified-Mobile.html`

> **v1.1 변경 이력** — v1.0을 실제 코드(3개 `<script>` 블록, 7,155줄) 재검증 기반으로 교차검토했습니다. 주요 수정 사항:
> 1. **네트워크 부하 모델 재산정** — `tick`(1.5초)·`computeAIState`(5초)·`updateSmcSignalOutcomes`(30초)는 코드 확인 결과 **네트워크 호출이 전혀 없는 순수 로컬 렌더/재계산 루프**였습니다. 실제 주기적 REST 폴링은 OI·L/S비율·USDT.D 세 갈래(60초 전후)뿐이라 원래 문서가 시사한 것보다 부하가 훨씬 낮습니다.
> 2. **"Worker" 용어 충돌 발견** — 기존 코드가 이미 브라우저 백그라운드 스레드(`smcWorker`, Web Worker API)를 "Worker"라 부르고 있어, Cloudflare Workers와 이름이 겹칩니다. 이 문서에서는 이후 **"CF Worker"**로 명시 구분합니다.
> 3. **WebSocket 프록시 판단 근거 수정** — 2025년 4월부터 Durable Objects가 Workers 무료 플랜에도 열려 있어 "무료라서 못한다"는 원래 근거는 틀렸습니다. 다만 이 대시보드는 단일 사용자·저지연 실시간 틱이 핵심이라 **가능해도 안 하는 쪽이 낫다**는 결론은 유지하되 이유를 바로잡았습니다 (3장·7장 참고).
> 4. **지역 차단 회피 주장 철회** — CF Worker는 기본적으로 "요청이 들어온 곳과 가까운" 위치에서 실행되므로, 접속 지역이 막혀 있다면 프록시를 붙여도 자동으로 우회되지 않습니다. 원문서가 이 문제를 해결한다고 암시한 부분은 근거가 부족해 삭제했습니다.
> 5. **누락 엔드포인트 추가** — CoinGecko `/api/v3/global` (USDT.D 지표) 호출이 원래 목록에서 빠져 있었습니다. 데모 API 키 변수(`COINGECKO_DEMO_API_KEY`)가 현재는 비어 있지만, 향후 값이 채워지면 클라이언트에 평문 노출되는 구조라 이것도 반영했습니다.
> 6. **기존 복원력 패턴 확인** — USDT.D 폴링에 이미 자기 재조정형 백오프(성공 시 60초, 3연속 실패 시 180초)가, L/S비율 호출에 이미 5초 디바운스 가드가 구현돼 있었습니다. 이 패턴들을 CF Worker 레이어로 일반화하는 방향으로 6장을 수정했습니다.
> 7. **우선순위 재조정** — 위 1번 결과로 "쿼터 보호용 캐싱"의 실익이 원래 생각보다 작다는 게 드러나, Access(접근 제어)를 캐싱보다 앞순위로 올렸습니다.

> **v1.2 변경 이력** — 실제 CF Worker(`cloudflare-worker/index.js`)와 대쉬보드 연동 코드를 만든 뒤, 그 결과물을 다시 감사한 내용을 반영했습니다:
> 1. **엔드포인트 통합** — `globalLongShortAccountRatio`와 `topLongShortPositionRatio`는 클라이언트의 같은 함수(`fetchFutsRest`)가 항상 같이 호출하길래, `/api/futures/lsratio` 하나로 합쳐 왕복을 1번으로 줄였습니다(부분 실패 허용은 유지).
> 2. **실제 버그 2건 발견·수정** — 제가 짠 코드를 다시 감사하다가 ① klines 라우트가 `Cache-Control` 헤더만 세팅하고 실제 Cache API(`caches.default`)에는 저장하지 않아 엣지 캐싱이 안 되고 있던 것 ② L/S+Top 통합 호출에서 `Promise.all`을 써서 한쪽이 네트워크 오류로 reject되면 의도와 달리 전체가 실패해버리던 것을 찾아 고쳤습니다(`Promise.allSettled`로 교체).
> 3. **용어 일관성 재점검** — 대쉬보드 쪽에 넣은 주석 중 일부가 "CF Worker" 대신 그냥 "Worker"라고만 써서 브라우저 자체 Worker(`smcWorker`)와 헷갈릴 수 있던 걸 전부 수정했습니다.
> 4. **Access 적용 범위 주의사항 추가** — 배포 시 Cloudflare Access를 메인 페이지에만 걸고 `/api/*`는 빠뜨리기 쉬운 지점이라 7장에 명시했습니다.

## 1. 현황 분석 (As-Is)

코드 분석 결과, 현재 구조는 **순수 클라이언트 사이드 SPA**입니다. 서버 없이 브라우저가 Binance(+CoinGecko)에 직접 요청합니다. 다만 겉보기와 달리 **하나의 통합 앱이 아니라, 서로 거의 독립적으로 동작하는 2개 모듈이 한 파일(`<script>` 3블록)에 합쳐진 구조**입니다 — 각 모듈이 `FAPI` 상수, `pollOI()` 함수 등을 각자 따로 선언합니다(실행 스코프가 분리돼 있어 충돌은 없음). 이는 문제라기보다 "동일 심볼 데이터를 두 모듈이 각자 중복 호출할 가능성"이라는 캐싱 설계상의 실질적 포인트입니다.

| 구분 | 내용 |
|---|---|
| 호스팅 | 단일 HTML 파일, 내부적으로 독립된 2개 모듈(SMC 엔진 / GAUGE-FLOW) 결합 (정적 배포처 불명 / 로컬 실행 추정) |
| 데이터 소스 | `fapi.binance.com`(선물), `api.binance.com`/`api1`/`api2`(현물, Binance 공식 미러 목록), `api.coingecko.com`(USDT.D) |
| REST 호출 | Kline, Premium Index, Global L/S Ratio, Top Trader Position Ratio, Open Interest, Depth, ExchangeInfo, CoinGecko Global — 총 8종 (호출 지점 기준 15곳) |
| 실제 주기적 폴링 | OI 60초(브라우저 백그라운드 스레드, 활성 심볼 최대 2개) · L/S+TopLS 60초(+5초 디바운스 가드 기존 구현됨) · USDT.D 60초(3연속 실패 시 180초로 자동 완화, 기존 구현됨) |
| 비주기·온디맨드 호출 | `premiumIndex`/`exchangeInfo`는 심볼 로드·전환 시 1회성 검증 호출일 뿐 반복 폴링 아님 |
| 순수 로컬(네트워크 없음) | `tick`(1.5초), `computeAIState`(5초), `updateSmcSignalOutcomes`(30초) — 코드 주석에도 "신규 fetch 트리거 없음"으로 명시돼 있음 |
| WebSocket | 4곳에서 직접 연결 — 실시간 캔들/틱/마크가격/오더북은 대부분 WS로 흐르고, REST는 초기 백필·재연결 후 갭 채우기 용도 |
| 문제점 | ① 2개 모듈이 같은 심볼을 동시에 볼 때 OI·klines 등을 중복 호출할 수 있음(빈도 자체는 낮아 레이트리밋보다는 "낭비" 성격) ② 페일오버·백오프 로직이 함수마다 제각각(USDT.D만 백오프 있고 OI는 없음) ③ 정적 파일 자체의 CDN·캐싱·HTTPS가 없어 모바일 최초 로딩이 느릴 수 있음 ④ 공개 URL로 노출 시 개인 트레이딩 로직·(향후 추가 시) CoinGecko 키가 그대로 노출됨 |

## 2. 최적화 목표

1. **비용 $0** — Cloudflare 무료 티어 범위 내에서만 구성
2. 응답 지연 단축과 중복 호출 제거 (엣지 캐싱) — Binance 레이트리밋은 재산정 결과 여유가 커서 부가적 목표로 하향
3. 정적 자산의 글로벌 CDN 배포로 모바일 초기 로딩 속도 개선
4. 개인용 트레이딩 도구이므로 **비인가 접근 차단** (실질적으로 가장 시급한 목표)
5. 미러 페일오버·백오프 로직을 엣지(CF Worker)로 이관하여 클라이언트 코드 단순화

## 4. Cloudflare 무료 컴포넌트별 활용 방안

### 4.1 Cloudflare Pages — 정적 호스팅
- `AMOS-Unified-Mobile.html`을 그대로 배포. 빌드 불필요(정적 파일이므로 `git push` 또는 대시보드 업로드만으로 배포).
- 정적 자산 요청은 **무료·무제한**이며 Workers 요청 쿼터를 소모하지 않음.
- 월 500회 빌드 한도 → 개인 프로젝트에는 충분.
- 커스텀 도메인 + 자동 HTTPS 무료 제공.

### 4.2 CF Worker — API 프록시 & 캐시
- 현재 클라이언트에 있는 `SPOTS=['api.binance.com','api1...','api2...']` 미러 배열(Binance 공식 로드밸런싱용 도메인)을 CF Worker 내부로 이관하여 `/api/futures/oi`, `/api/futures/lsratio`, `/api/usdtd`, `/api/futures/premium`, `/api/spot/exchangeinfo`, `/api/spot/klines`, `/api/futures/klines`, `/api/futures/depth` 라우트로 통합(실제 구현은 `cloudflare-worker/index.js` 참고).
- **무료 한도(2026년 기준)**: 요청 100,000회/일(UTC 자정 리셋), 요청당 CPU 10ms, 요청당 서브리퀘스트 최대 50개. 단순 fetch 프록시는 CPU 사용량이 매우 낮아(평균 2~3ms) 이 한도 안에서 넉넉히 동작.
- **실제 요청량 재산정**: 코드 확인 결과 주기적 REST 폴링은 OI(60초, 활성 심볼 최대 2개 ≈ 2,880회/일) + L/S+TopLS(60초 ≈ 1,440회/일) + USDT.D(60초 ≈ 1,440회/일) 뿐이며, `premiumIndex`/`exchangeInfo`는 심볼 로드 시 1회성이라 무시 가능한 수준입니다. **무캐싱 상태에서도 하루 5,000~7,000건 정도로, 10만 건 한도의 5~7%**에 불과합니다.
- 이 여유를 감안하면 캐싱의 주 목적은 "쿼터 보호"가 아니라 **① 2개 모듈이 같은 심볼을 볼 때의 중복 호출 제거, ② 응답 지연 단축, ③ 엔드포인트별로 제각각이던 재시도/백오프 로직 통일**입니다. 여러 탭·기기를 동시에 켜두는 사용 패턴이 생기면 그때부터 캐싱의 쿼터 보호 효과가 본격적으로 의미를 가집니다.
- Cache API(`caches.default`)로 Worker 자체에서 응답을 엣지 캐싱하고, 기존 코드에 이미 있던 USDT.D의 자기 재조정 백오프(60→180초)와 L/S 호출의 5초 디바운스 가드를 모든 라우트에 공통 적용.

### 4.3 Workers KV — 이번 설계에서는 생략 권장
- 원래 `exchangeInfo` 캐싱용으로 검토했으나, 실제로는 심볼을 수동으로 로드/전환할 때만(하루 몇 번 수준) 호출되는 온디맨드 값이라 별도 스토리지(KV)를 둘 실익이 없습니다.
- Cache API만으로 충분하고, KV를 추가하면 바인딩 설정·쓰기 쿼터 관리라는 복잡도만 늘어납니다. 나중에 여러 기기 간 설정 공유처럼 "쓰기가 드물고 여러 클라이언트가 같이 읽어야 하는" 요구가 생기면 그때 도입을 재검토하면 됩니다.

### 4.4 Cloudflare Access (Zero Trust) — 접근 제어
- 무료 플랜에서 최대 50명까지 이메일 OTP/Google 로그인 기반 접근 제어 제공.
- 개인 트레이딩 대쉬보드이므로 Pages 도메인 전체에 Access 정책을 걸어 URL을 아는 제3자의 무단 접근을 차단.

### 4.5 Cache Rules / Browser Cache TTL
- HTML은 짧은 TTL(변경 반영 목적), 폰트·CSS 등 정적 리소스는 긴 TTL로 설정해 모바일 재방문 시 로딩 속도 개선.

### 4.6 Cloudflare Web Analytics
- 별도 스크립트 삽입만으로 무료 실사용자 성능 지표(로딩 시간, 지역별 접속) 확인 가능. 쿠키 미사용 방식.

### 4.7 기본 WAF / Bot Fight Mode
- 무료 플랜 기본 포함 규칙만으로도 URL 유출 시 자동화된 스크래핑·과도한 트래픽을 1차 방어.

## 5. 엔드포인트별 캐싱 전략 (Worker Cache TTL 제안)

| 엔드포인트 | 실제 호출 패턴 | 데이터 특성 | 제안 캐시 TTL |
|---|---|---|---|
| `klines` (limit 500, 백필) | 최초 로딩/재연결 시 | 과거 완결봉은 불변 | 완결봉 구간은 장기 캐시, 마지막 진행 중 봉만 짧게 |
| `klines` (limit 10, 갭 채우기) | WS 재연결 후 | 자주 변함 | 3~5초 |
| `premiumIndex` | 심볼 로드 시 1회성 | 초 단위 변동 | 캐싱 실익 적음 — 통과 처리 |
| `exchangeInfo` | 심볼 로드 시 1회성 | 거의 고정 | 짧게 캐시(수 분)해 두 모듈이 같은 심볼을 동시에 로드할 때만 중복 제거 |
| `lsratio` (global+top 통합) | 60초 폴링(+5초 디바운스 기존 구현), `/api/futures/lsratio` 1회 호출로 통합 | 5분 단위 데이터 | 30~60초 |
| `openInterest` | 60초 폴링, 활성 심볼 최대 2개 | 완만한 변동 | 30~60초 |
| `coingecko/global` (USDT.D) | 60→180초 자기 백오프(기존 구현) | 완만한 변동 | 30~60초, 백오프 로직은 CF Worker에서도 유지 |
| `depth` | 필요 시(WS 폴백) | 실시간성 중요 | 캐시 없음(패스스루) |

TTL은 Worker의 `Cache-Control` 헤더 + `caches.default.put()`으로 구현합니다. 다만 위 4.2절에서 보듯 이 대시보드의 실제 호출량은 쿼터 여유가 크므로, 캐싱의 체감 효과는 "레이트리밋 회피"보다 "두 모듈의 중복 호출 제거"와 "지연시간 단축" 쪽에서 더 크게 나타납니다.

## 6. 단계별 구현 로드맵

**1단계 — 정적 호스팅 이전 (당장 가능)**
- Cloudflare Pages에 `AMOS-Unified-Mobile.html` 배포, 커스텀 도메인 연결.
- 코드 변경 없이도 CDN·HTTPS 이점 확보.

**2단계 — 접근 제어 (원안보다 순서를 앞당김)**
- 실제 요청량 재산정 결과(4.2절) 쿼터 위험이 낮다는 게 확인된 반면, 개인 트레이딩 로직이 담긴 URL을 그대로 인터넷에 노출하는 리스크는 그대로 남아 있어, Access를 캐싱보다 먼저 적용하는 편이 위험 대비 효과가 큽니다.
- Cloudflare Access로 대쉬보드 도메인 보호(이메일 OTP 등).

**3단계 — API 프록시 CF Worker 도입**
- REST 호출부(8종 엔드포인트, 15개 호출 지점)를 자체 CF Worker 라우트로 교체.
- 미러 페일오버·TTL 캐싱·백오프/디바운스 로직을 CF Worker로 이관해 클라이언트 코드 단순화. KV는 도입하지 않음(4.3절).
- WebSocket은 이관하지 않고 브라우저→Binance 직접 연결 유지(3장 근거 참고).

**4단계 — 모니터링 및 튜닝**
- Web Analytics로 실사용 패턴 확인 → 캐시 히트율과 두 모듈 간 중복 호출 감소폭을 관찰. 하루 요청량이 예상(5~7천 건)보다 뚜렷이 높게 나오면 그 시점에 원인(다중 기기·탭)을 확인.

## 7. 무료 티어 한도 및 주의사항

- CF Worker 무료: **100,000 요청/일**, 요청당 CPU 10ms, 서브리퀘스트 최대 50개/호출 — 초과 시 에러(1027) 또는 스로틀. 이 대시보드의 실제 추정 부하(5~7천 건/일, 4.2절)는 이 한도의 5~7% 수준이라 여유가 큽니다.
- **WebSocket**: Durable Objects가 2025년 4월부터 무료 플랜에서도 생성 가능해져 "비용 때문에 못 한다"는 더 이상 사실이 아닙니다. 다만 1인 사용·저지연이 핵심인 이 대시보드에서는 굳이 프록시를 거치지 않는 편이 낫다는 게 이 문서의 결론입니다 — 프록시는 다중 클라이언트가 업스트림 연결 하나를 공유할 때 이득이지, 단일 사용자 릴레이는 홉만 추가됩니다. CF Worker 자체(Durable Objects 없이)도 WebSocket 업그레이드를 무료 플랜에서 처리할 수 있다는 점은 참고로 남겨둡니다.
- Pages 빌드는 월 500회 — 개인 프로젝트 배포 빈도로는 문제 없음.
- CF Worker는 기본적으로 요청이 들어온 곳과 가까운 위치에서 실행됩니다. 접속 지역·네트워크 문제로 Binance 접속 자체가 막히는 상황이라면, 프록시를 붙이는 것만으로 자동 해결되지는 않는다는 점을 알아두시기 바랍니다.
- 여러 기기/탭을 동시에 오래 켜두는 사용 패턴이 생기면 CF Worker 요청량이 추정치보다 빨리 늘 수 있으므로, 캐시 히트율을 Analytics로 주기적 확인 권장.
- **Access 적용 범위 확인 필수**: Access 정책을 메인 페이지 경로에만 걸고 `/api/*`(CF Worker 라우트)를 빠뜨리면, 대쉬보드 화면은 로그인으로 막혀 있어도 API 프록시 자체는 그대로 공개돼 있는 상태가 될 수 있습니다. Access Application을 설정할 때 도메인 전체(`your-domain.com/*`)를 대상으로 하거나, `/api/*`를 별도 Application으로 추가해서 명시적으로 포함시켜야 합니다.
